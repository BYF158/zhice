# 第一部分：软件测试计划

## 1. 引言

### 1.1 编写目的

本文档描述智测心理测试系统的测试计划，明确测试目标、范围、策略、资源、进度等，指导测试工作的开展。

### 1.2 背景

本系统基于若依框架开发，需要进行全面的测试以确保系统质量。

### 1.3 定义

| 术语 | 定义 |
|-----|------|
| 单元测试 | 对软件中最小可测试单元进行检查和验证 |
| 集成测试 | 将多个单元组合起来进行测试 |
| 系统测试 | 对整个系统进行功能和性能测试 |
| 验收测试 | 用户确认系统是否满足需求的测试 |

## 2. 测试目标

### 2.1 总体目标

- 确保系统功能满足需求规格说明书的要求
- 确保系统性能满足性能指标
- 确保系统安全符合安全标准
- 发现并修复系统缺陷
- 为系统上线提供质量保证

### 2.2 具体目标

- 功能测试覆盖率 ≥ 90%
- 关键业务场景测试覆盖率 100%
- 性能测试通过率 100%
- 安全测试无高危漏洞
- 缺陷修复率 100%（严重和重要级别）

## 3. 测试范围

### 3.1 功能测试

| 模块 | 测试内容 | 优先级 |
|-----|---------|-------|
| 用户管理 | 登录、注册、修改个人信息、修改密码 | P0 |
| 权限管理 | 角色管理、菜单管理、权限控制 | P1 |
| 人格测试 | 加载题目、提交答案、查看结果 | P0 |
| 题目管理 | 增删改查、批量操作、导出 | P1 |
| 结果管理 | 查询结果、导出结果、统计分析 | P1 |
| 系统管理 | 字典管理、参数管理、日志管理 | P2 |

### 3.2 性能测试

| 测试项 | 测试指标 | 目标值 |
|-------|---------|-------|
| 响应时间 | 页面加载 | < 2秒 |
| 响应时间 | API调用 | < 500ms |
| 并发能力 | 并发用户数 | ≥ 1000 |
| 吞吐量 | QPS | ≥ 500 |

### 3.3 安全测试

### 3.4 兼容性测试

| 测试项 | 测试内容 |
|-------|---------|

## 4. 测试策略

### 4.1 测试类型

#### 4.1.1 单元测试

- **工具**：JUnit、Mockito
- **范围**：Service层、工具类
- **覆盖率目标**：≥ 70%

#### 4.1.2 集成测试

- **工具**：Spring Boot Test、Postman
- **范围**：API接口
- **覆盖率目标**：≥ 80%

#### 4.1.3 系统测试

- **工具**：Selenium、JMeter
- **范围**：完整业务流程
- **覆盖率目标**：100%（关键流程）

#### 4.1.4 性能测试

- **工具**：JMeter、LoadRunner
- **范围**：关键接口、页面加载
- **场景**：正常负载、高峰负载、极限负载

#### 4.1.5 安全测试

- **工具**：OWASP ZAP、Burp Suite
- **范围**：全部接口、用户输入
- **标准**：OWASP Top 10

### 4.2 测试方法

| 测试方法 | 说明 |
|---------|------|
| 黑盒测试 | 不关注内部实现，只测试功能 |
| 白盒测试 | 关注代码逻辑，进行代码走查 |
| 灰盒测试 | 结合黑盒和白盒测试 |

### 4.3 缺陷管理

- **缺陷等级**：严重、重要、一般、轻微
- **缺陷状态**：新建、已分配、已修复、已验证、已关闭
- **缺陷生命周期**：新建 → 已分配 → 已修复 → 已验证 → 已关闭

### 7.1 风险识别

| 风险 | 影响 | 概率 | 应对措施 |

| 需求变更 | 高 | 中 | 及时更新测试用例 |

- **人员不足**：提前规划，必要时申请外部资源
- **环境不稳定**：准备多套测试环境，建立环境监控
- **缺陷修复延迟**：制定缺陷优先级，确保核心功能先测试

| 复现步骤 | 1. 完成所有题目作答<br>2. 点击提交按钮<br>3. 观察页面跳转情况 |
